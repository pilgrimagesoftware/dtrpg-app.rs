## Context

Both `services/sdk.rs` (1203 lines) and `services/collections_sdk.rs` (765 lines) follow the same internal shape: a gateway trait defining the SDK operation boundary, a real HTTP-backed gateway implementation, a stub/unavailable gateway for the unauthenticated state, a `*Service` struct implementing the app-facing `LibraryService`/`CollectionsService` trait by delegating to the gateway, a handful of free functions translating SDK/HTTP errors into the service's own error type, and (in `sdk.rs` only) response-shape-to-domain-model mapping logic. Tests for all of the above currently live in one trailing `mod tests` per file.

Crucially, both files' HTTP gateways are also near-duplicates of each other at the connection level: both wrap the same `dtrpg_sdk::LibraryClient` type (it serves the whole DTRPG API surface, not just "library" endpoints — collections operations like `list_product_lists` are also methods on it), and both build that client via an identical sequence — resolve the API key from the keyring or environment, read/accept access and refresh tokens, construct `Config`/`DriveThruRpgSdk`, apply the auth response, fetch the client, and spin up a Tokio `Runtime`. Only the final error-wrapping type differs (`LibraryServiceError` vs `CollectionsServiceError`), and even most of the error *wording* is identical modulo the words "library"/"collections".

`crates/dtrpg-core/CLAUDE.md`-inherited project style (docs/rust.md) caps files at 700 lines and functions at 50 lines, and specifies unit tests live colocated with the code they test, not in a single trailing block.

## Goals / Non-Goals

**Goals:**
- Consolidate both files under a single `services/sdk/` module (not two parallel top-level modules), organized by domain submodule (`library`, `collections`).
- Extract the genuinely shared connection-building logic into a peer module (`sdk/connection.rs`) rather than duplicating it per domain.
- Bring every file under the 700-line cap.
- Bring `map_order_product` under (or close to) the 50-line function guideline via extraction, without changing its output for any existing test case.
- Colocate tests with the code they exercise.
- Zero behavior change: same public types, same runtime behavior, same test outcomes. `RustSdkCollectionsService`'s import path does change (`services::collections_sdk::` -> `services::sdk::`), since the whole point is collapsing it into `sdk`; its one call site is updated as part of this change.

**Non-Goals:**
- Changing the `SdkLibraryGateway`/`SdkCollectionsGateway` trait shapes, the `LibraryService`/`CollectionsService` trait impls' method signatures, or any error classification logic.
- Renaming `RustSdkLibraryService`/`RustSdkCollectionsService`.
- Touching `login.rs` (105 lines, already appropriately sized) or anything outside `crates/dtrpg-core/src/services/` and its one external call site in `app/mod.rs`.
- Sharing `map_client_error` between domains: `LibraryServiceErrorKind` has a `NeedsReauth` variant that `CollectionsServiceErrorKind` doesn't, and collections has a 409-Conflict special case library doesn't — the HTTP-status-to-kind classification genuinely differs, not just wording, so this stays domain-specific.
- Sharing a gateway trait between the two services — `SdkLibraryGateway`'s pagination-oriented operations and `SdkCollectionsGateway`'s membership-oriented operations don't share a meaningful common interface; only the underlying *transport setup* (client + runtime) is shared, which is exactly what `connection.rs` captures.

## Decisions

- **Domain submodules (`sdk::library`, `sdk::collections`), each a self-contained gateway/service/error trio, with `connection.rs` as a peer.** This was an explicit correction mid-implementation: the first pass split `sdk.rs` and `collections_sdk.rs` into two separate top-level module directories (`services::sdk::{gateway,mapping,errors}` and `services::collections_sdk::{gateway,errors}`), each internally organized but with no relationship to each other. That didn't reflect that they're one capability (SDK-backed services) with two domains, and it duplicated the connection-building logic verbatim. Consolidating under one `sdk` module with `library`/`collections` as its two domain submodules, and hoisting the actually-shared connection logic to `sdk::connection` as a sibling of both, matches how the crate already organizes multi-concern features (e.g. `dtrpg-ui`'s `ui/<feature>/{state,data}.rs`) and avoids the duplication.
- **`sdk/connection.rs` is domain-agnostic; each domain's `errors.rs` translates its outcome.** `connection.rs` returns a neutral `ConnectionError` enum (`MissingApiKey`, `MissingApplicationKeyEnv`, `MissingAccessTokenEnv`, `Sdk(SdkError)`, `RuntimeInit(io::Error)`) rather than either service's error type. Each domain's `errors.rs` gets a `map_connection_error` function that matches on `ConnectionError` and produces its own `LibraryServiceError`/`CollectionsServiceError` with its own exact wording (e.g. "...to load SDK-backed library data." vs "...collections data."). This keeps `connection.rs` reusable without forcing a fake-shared error type or losing the small per-domain wording differences that already existed.
  - Alternative considered: have `connection.rs` take a generic error-mapping closure or `From` bound. Rejected as unnecessary indirection for two call sites — a plain match in each `errors.rs` is more readable and just as short.
  - Alternative considered: keep `map_sdk_error` merged into `connection`'s error translation entirely. Rejected because `map_sdk_error` is also called directly from each domain's `map_client_error` (for the `ClientError::Sdk(...)` variant), so it has to exist standalone in each domain's `errors.rs` regardless; `map_connection_error` just delegates to it for the `ConnectionError::Sdk` case rather than duplicating its match arms.
- **`sdk/mod.rs` re-exports both service types**, so external callers use one consistent path (`crate::services::sdk::RustSdkLibraryService`, `crate::services::sdk::RustSdkCollectionsService`) without needing to know about the `library`/`collections` submodule split — matching "everything SDK-related lives under one `sdk` module."
- **`mapping.rs` only exists for `library`, not `collections`.** `collections`'s gateway calls already return data close to what `CollectionsService` needs; only error mapping and the two JSON field-extraction helpers (`extract_member_id`, `extract_product_list_item_id`, kept in `collections/mod.rs` since both the service impl and gateway need them) require any transform logic. Forcing an empty or trivial `mapping.rs` there would be a distinction without a difference.
- **`map_order_product` decomposition boundaries follow its existing comment-delimited sections, not an arbitrary line-count split**: extracted `resolve_publisher`, `resolve_product_info`/`resolve_cover_url`, `resolve_kind`, `resolve_format`, `map_files` (the dedup + per-file mapping), and `resolve_year`. `map_order_product` itself becomes an orchestrator calling each in sequence and assembling the final `LibraryItem` — 54 lines, slightly over the 50-line guideline, but ~22 of those are an unavoidable one-field-per-line struct literal; not worth an artificial further split.
- **Tests move with their subject, not into a shared `tests.rs`.** `library/mapping.rs` and `library/mod.rs` each get their own `#[cfg(test)] mod tests`; `collections/mod.rs` gets one covering both the service-level tests and the two extraction-function unit tests, since both live in that file. `FakeSdkGateway`/`order_product_item`/`pagination_links` are duplicated between `library/mapping.rs`'s and `library/mod.rs`'s test modules since both genuinely need them (per this repo's reuse-before-duplicating threshold of "more than two" call sites; two is the accepted duplication case here, per this design's own precedent for the two connection-error messages).

## Risks / Trade-offs

- [Moving code across files risks a mechanical error — a missed `use`, a dropped `pub(super)`, or a test accidentally not carried over] → Mitigation: `cargo check --workspace --all-targets` and `cargo test --workspace` must both pass with identical test counts (before/after) before this change is considered done.
- [Decomposing `map_order_product` into six-plus helper functions could obscure the original function's implicit ordering dependencies, if any exist] → Mitigation: read through the existing function fully before extracting — the transforms are independent (publisher/cover/kind/format/files/year each read only from `attributes`/`item`/lookup maps, none depend on another's output), so extraction is safe; the existing tests (`map_order_product_*`) pin the combined behavior regardless.
- [Extracting `connection.rs` risks subtly changing which error variant/kind a given failure maps to, since the logic moved through an intermediate `ConnectionError` layer] → Mitigation: `map_connection_error` in each domain reproduces the exact same kind/message pairing the original inline code had, verified by keeping both domains' error tests green (`sdk_service_preserves_session_error_classification`, `session_error_propagates`, etc. all still pass unchanged).
- [Consolidating under one `sdk` module changes `RustSdkCollectionsService`'s import path] → Mitigation: only one call site exists (`app/mod.rs`), grepped and updated directly; `cargo check --workspace` confirms no other reference was missed.
